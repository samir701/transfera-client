    main() & process lifecycle

    Entry point: server/src/main_runner.cpp. One OS process, no fork() of the API.

    1. Full main() source flow

    int main() {
    const int port = readPortFromEnv();   // default 8080

    std::signal(SIGINT, onSignal);
    std::signal(SIGTERM, onSignal);

    FileController controller(port);
    if (!controller.start()) return 1;

    while (g_keepRunning)
        std::this_thread::sleep_for(std::chrono::milliseconds(200));

    controller.stop();
    return 0;
}

    2. Signal handling

    std::atomic<bool> g_keepRunning{true};

void onSignal(int) { g_keepRunning = false; }
    server/src/main_runner.cpp:12-17

    SIGINT / SIGTERM do not call stop() inside the handler — only flip a flag. Actual teardown runs on the main thread after the loop.

    3. The sleep loop (why it exists)

    QuestionAnswer
    Why not block in main?HTTP listen runs on serverThread_
    Why sleep?Avoid 100% CPU busy-wait while waiting for shutdown
    Why 200 ms?Human-scale shutdown latency; not tied to I/O

    4. Process spawn map

    OS Process: ./build/server (single PID) │ ├─ Thread: main │ └─ sleep until g_keepRunning == false → controller.stop() │ ├─ Thread: serverThread_ (FileController) │ └─ httplib listen loop → accept → enqueue to ThreadPool │ ├─ Thread(s): httplib ThreadPool workers (N ≈ CPU count) │ └─ handleUpload / handleDownload / health │ └─ Thread(s): detached std::thread per upload └─ FileSharer::startFileServer(port) — blocks until one client served

    5. Child processes?

    None for the API. The project does not call fork(), exec(), or system() for serving. “Child” in conversation sometimes means detached threads — still same process, same address space.
