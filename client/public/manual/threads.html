    2. Detached upload thread (critical)

    std::thread([this, port]() {
    fileSharer_.startFileServer(port);
}).detach();
    server/src/services/fileController.cpp:308-310

    Why detach?

    • handleUpload must return HTTP 200 immediately with {"port": N}.
    • startFileServer blocks on accept() — may wait seconds for downloader.
    • Joining in the handler would stall the httplib worker thread for the whole wait.

    Risks

    • No exception propagation to upload handler.
    • Lifetime: captures this → FileController must outlive the thread (true until process exit).
    • Many uploads → many concurrent detached threads + TCP listeners.

    3. Atomics: running_ and listen_ok_

    if (running_.exchange(true)) return listen_ok_.load();
listen_ok_ = true;  // set inside serverThread_ after bind
    server/src/services/fileController.cpp:104-121

    exchange prevents double-start races between main and destructor paths.

    4. Zombie threads vs zombie processes

    Zombie process (Unix): Child exited but parent didn't wait() — shows as Z in ps. This project: no child processes for serving.
    “Zombie” colloquially: Detached threads still running after you think shutdown completed. If uploads keep spawning P2P servers, ports stay open until each thread finishes. Killing the process with kill -9 terminates all threads immediately.
