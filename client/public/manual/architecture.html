    System architecture

    HTTP control plane (fixed API port) + P2P data plane (socketpair per download; invite port is a catalog key only).

    1. Architectural layers

    LayerTechnologyResponsibility
    PresentationNext.js (GitHub Pages)Upload validation, max downloads UI, invite display
    APIcpp-httplib on RenderREST endpoints, CORS, multipart
    CatalogFileSharer::available_files_Invite → path, name, maxDownloads, downloadCount
    Transfersocketpair + P2P framingPer-download sender/receiver threads
    Storage$TMPDIR/transfera-uploads/Until quota consumed or process restart

    2. Port topology

    Port 8080 / Render PORT — HTTP API (health, upload, download) Port 49152–65534 — Invite codes only (not TCP bind for HTTP path) Timeline (maxDownloads = 2): T0 POST /api/upload → {port: 58341, maxDownloads: 2} T1 GET /api/download/58341 → socketpair P2P → remaining=1 T2 GET /api/download/58341 → socketpair P2P → remaining=0, invite removed T3 GET /api/download/58341 → 404

    3. Why HTTP + P2P framing?

    Browsers use GET + blob for downloads. Bulk bytes use the legacy Filename:\n + binary protocol over a private socketpair inside the API process — same wire format as the original TCP design, without upload-time listeners or localhost port races.

    5. End-to-end sequence

    1. Client validates file (5 MB, no video) + sends maxDownloads
    2. offerFile registers invite; file on disk
    3. Downloader calls GET /api/download/{invite}
    4. receiveViaLocalP2P runs P2P; recordSuccessfulDownload updates quota
    5. When quota exhausted → consumeInvite deletes catalog entry + file
    Render free tier: in-memory invites and /tmp files are lost on cold start or redeploy. Upload and all downloads must hit the same running instance.
