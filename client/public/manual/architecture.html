    System architecture

    Two network planes: HTTP control plane (port 8080) and TCP data plane (dynamic ports 49152–65534).

    2. Port topology

    Port 8080 (fixed) — HTTP API (always) Port 49152–65534 — Ephemeral P2P (one per uploaded file, random) Example timeline: T0 POST /api/upload → returns {"port": 58341} T1 Thread detached → TCP bind 0.0.0.0:58341, listen(1) T2 GET /api/download/58341 → HTTP handler connect 127.0.0.1:58341 T3 TCP sends "Filename: doc.pdf\n" + bytes T4 HTTP responds with attachment to browser

    3. Why two protocols?

    Browsers cannot safely expose raw TCP to JavaScript for arbitrary peer transfer. The design mirrors the original Java version: HTTP for orchestration, TCP for bulk bytes. The HTTP download handler acts as a local proxy (connect(127.0.0.1, peerPort)) so the browser still uses familiar GET + blob download.

    5. Sequence: end-to-end share

    1. User drops file → FormData POST to /api/upload
    2. FileController writes disk, calls offerFile → random free port
    3. detach() starts startFileServer(port)
    4. UI shows invite port
    5. Receiver enters port → GET /api/download/{port}
    6. HTTP worker connects TCP, reads file, returns Content-Disposition
    Not true P2P across the Internet: TCP servers bind locally; download proxy uses 127.0.0.1. Remote peers would need the host IP and firewall rules — out of scope for current code.
