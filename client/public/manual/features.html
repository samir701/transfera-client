    Product features

    Current behavior of Transfera (UI + C++ API) as of the latest release. This page is the operator-facing summary; other chapters explain implementation detail.

    P2P download path

    Each download runs the P2P wire protocol (Filename: header + binary stream) over a coordinated socketpair inside the API process. No TCP listener is started on upload.

    Max downloads (N)

    Uploader sets how many successful downloads are allowed (default 1, range 1–100). After the Nth success the invite is removed and the temp file is deleted.

    Upload limits

    Frontend enforces 5 MB max size and blocks video files (MIME video/* and common extensions such as MP4, MOV, WebM).

    Invite code

    A random port number in 49152–65534 is the shareable invite code. It is only a lookup key; the actual P2P channel uses an ephemeral socket pair per download request.

    Production hosting

    UI on GitHub Pages, API on Render (Docker). HTTPS API URL is baked into the static client build.

    Observability

    Structured [INFO] / [ERROR] logs to stdout; optional HTTP access log via TRANSFERA_LOG_HTTP and verbose P2P lines via TRANSFERA_LOG_VERBOSE.

    1. Upload (Share a File)

    RuleWhereDetail
    Max file size Browser (dropzone) 5 MB — client/src/lib/uploadValidation.ts
    No videos Browser Rejected by MIME and extension (MP4, MOV, MKV, WebM, …)
    Max downloads UI tab row + multipart Field maxDownloads, default 1, sent with file
    Storage API server $TMPDIR/transfera-uploads/{hash}_{name}
    Response API {"port": <invite>, "maxDownloads": N}

    2. Download (Receive a File)

    1. User enters invite code → GET /api/download/{port}
    2. API claims invite (rejects if expired, exhausted, or transfer in progress)
    3. P2P transfer via receiveViaLocalP2P (socketpair + sender thread)
    4. On success: increment download count; if count ≥ maxDownloads → remove invite + delete file
    5. HTTP response: attachment body + X-Downloads-Remaining header
    Failed downloads do not consume a slot — the invite stays valid until N successful transfers. Concurrent downloads on the same invite return 404 with “download already in progress”.

    MethodPathBody / paramsResponse
